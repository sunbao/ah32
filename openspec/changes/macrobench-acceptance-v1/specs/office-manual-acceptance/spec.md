# Spec — office-manual-acceptance (v1)

## 一句话说明

这份规范定义“人怎么验收”：每轮迭代至少要在 Writer / ET / WPP 各跑一遍关键流程，确保**能聊、能写回、不乱写、出错能定位**。

## 适用范围

- 宿主：Writer（`host_app=wps`）、ET（`host_app=et`）、WPP（`host_app=wpp`）
- 部署：
  - 本机后端（前后端同机）
  - 远端后端（WPS 插件连接云端/局域网后端）

## 总规则（最关键的几条）

### Requirement: 每轮至少各跑一次（真环境）

- 每轮迭代至少在 **Writer / ET / WPP 各跑 1 次**完整链路：
  - 发起对话 → 流式返回 → 生成可执行 Plan → 执行写回 → 文档出现结果
- 目的：避免“某个宿主早就坏了，但没人发现”。

### Requirement: 多文档并发不串台

- 同时打开 **2 个 Writer + 1 个 ET + 1 个 WPP**，并发发起对话与写回：
  - 对话可以同时流式返回（互不阻塞）
  - 写回进入各自队列（同一时刻只执行 1 个写回，但不会写错文档）

### Requirement: 切换文档不“卡住/串会话”

- 当文档 A 正在生成中（还在流式输出）时，切换到文档 B：
  - 文档 B 的聊天历史要能立刻出现（不被“正在生成”挡住）
  - 文档 B 仍然可以继续发起对话（互不阻塞）
- 目的：避免“切文档后一直显示思考中/取消按钮/无法发送”的问题。

### Requirement: 写回队列要可控（能取消、能看到状态）

- 用户至少能做到：
  - 取消当前会话的生成
  - 取消某个文档队列里“还没执行的写回”
  - 看到写回状态：等待中 / 写回中 / 成功 / 失败（失败要能点开看到原因）

### Requirement: 写回只允许 Plan JSON

- 写回执行器只接受 `ah32.plan.v1` 的 Plan JSON。
- 普通聊天内容只进会话，不得直接写进文档正文。
- 目的：避免把“后端内部 JSON/工具 JSON/调试内容”误写到文档里。

### Requirement: 出错要给用户看得懂的提示

- 出错时，用户至少能看到：
  - 发生了什么（例如“写回失败/插图失败/网络失败”）
  - 影响是什么（例如“本次没有写入文档”）
  - 下一步怎么做（例如“重试/截图反馈/查看错误详情”）
- 同时后端/前端要留下可追踪的日志（带 trace_id 最好）。

## 手工验收条目（v1 最小集合）

> 下面每条都写“步骤 + 期望结果 + 留证据”，不要求你一次跑全量；但标注为“每轮必跑”的要每轮跑。

## 证据模板（照着填就行）

> 目的：你验收完我能直接看懂“跑了什么、成了还是没成、失败去哪查”。

### 成功（最小证据）

- 宿主：Writer/ET/WPP（选一个）
- 部署：本机后端/远端后端（选一个）
- 做了什么：1 句话（例如“生成摘要并写回到光标处”）
- 结果截图：包含写回后的文档画面（最好能看到写回区块）
- trace_id（可选但强烈建议）：能从后端日志定位到这一轮

### 失败（最小证据）

- 宿主 + 部署：同上
- 复现步骤：1/2/3
- 前端看到的错误提示：原样贴出来或截图
- 后端日志片段：至少包含异常堆栈（最好带 trace_id/tenant_id/user_id）
- 影响：例如“本次没有写入文档/写回写错了文档/插件重载了”

### Scenario: Writer 基础写回（每轮必跑）

- 步骤：
  1. 打开一个 Writer 文档
  2. 发起一次对话（例如“帮我写一段摘要并插入到光标处”）
  3. 触发写回执行
- 期望结果：
  - 文档出现新增内容
  - 不出现 Plan JSON / 工具 JSON 这种“内部内容”
- 证据：
  - 成功截图（含写回后文档）
  - 若失败：截图 + 错误提示 + 后端日志片段

### Scenario: ET 基础写回（每轮必跑）

- 步骤：
  1. 打开一个 ET 表格（有一块数据）
  2. 发起一次对话（例如“帮我生成一个图表/透视表”）
  3. 触发写回执行
- 期望结果：
  - 表格出现图表/透视表/排序筛选结果之一（以 Plan 为准）
  - 不写错工作表/不写错文档
- 证据：同上

### Scenario: WPP 基础写回（每轮必跑）

- 步骤：
  1. 打开一个 WPP 演示文稿
  2. 发起一次对话（例如“生成 3 页汇报 PPT：封面/要点/总结”）
  3. 触发写回执行
- 期望结果：
  - 新增幻灯片与内容
  - 不出现卡死/白屏（允许少量 best-effort 降级，但要有可见结果）
- 证据：同上

### Scenario: Writer 排版（每轮建议跑）

> 对应 `TODOLIST_WRITER.md` 里的“文档排版”场景，验收点是：排版类操作能稳定执行，不会把整篇搞乱，失败也能定位。

- 步骤：
  1. 打开一个 Writer 文档（最好包含：标题/正文/表格）
  2. 发起对话（例如“帮我把这份文档排版得专业一点：统一标题样式、段落间距、表格样式”）
  3. 触发写回执行
- 期望结果：
  - 标题样式/段落格式/表格样式至少有一项发生了可见变化
  - 不会“整篇格式大爆炸”（可以 best-effort，但要尽量局部修改）
  - 失败时有清楚提示（本次没写入/哪里失败）
- 证据：按“成功/失败模板”留

### Scenario: Writer 润色（每轮建议跑）

> 对应 `TODOLIST_WRITER.md` 里的“内容润色”场景。

- 步骤：
  1. 选中一段文字或把光标放在目标段落附近
  2. 发起对话（例如“把这段话改得更专业一点”）
  3. 触发写回执行
- 期望结果：
  - 文档出现润色后的版本（插入/替换都可以，但要可见）
  - 不把内部内容写进文档
- 证据：按“成功/失败模板”留

### Scenario: Writer 写回回退（每轮建议跑）

> 目的：写回后可以一键回退到“写回前的上一版”，避免误写造成不可逆覆盖。

- 步骤：
  1. 在 Writer 文档里先写一段明显的原始内容（例如“原始版本 ABC”）
  2. 发起一次对话并触发写回（让它改写/补充这段内容）
  3. 写回成功后，在聊天消息的代码块下方点击“回退上一版”
- 期望结果：
  - 对应块内容恢复为写回前的文本（上一版）
  - 若没有备份（从未写回过/备份被清理），不应显示回退按钮；或点击后给出明确提示
- 证据：成功/失败截图（最好包含文档内容变化 + 提示信息）

### Scenario: Writer 答题/填空写回（里程碑必跑）

> 对应 `TODOLIST.md` 的“Answer Mode”。验收点是：能在原题干结构里把答案填进去，重复执行不会越写越乱。

- 步骤：
  1. 打开一份带题号/括号/横线的 Writer 文档（模拟试卷/问卷均可）
  2. 发起对话（例如“按题号 1-5 填空作答，答案写回到原题干的横线处”）
  3. 触发写回执行
  4. 再执行一次相同写回（验证幂等）
- 期望结果：
  - 答案写在题干对应位置（不是另起一篇新文档/新模板）
  - 重复执行不会把答案越填越乱（能稳定覆盖或识别已填）
  - 失败时能告诉用户“哪题没定位到/下一步怎么修”
- 证据：按“成功/失败模板”留

### Scenario: ET 典型分析（每轮建议跑）

> 对应 `TODOLIST_ET.md` 的典型场景（趋势/对比/占比）。验收点是：能对数据做一个“排序/汇总/出图”的闭环。

- 步骤：
  1. 打开一个 ET 表格（含日期/分类/数值列更好）
  2. 发起对话（任选其一）：
     - “帮我分析销售趋势，做一张折线图并加趋势线”
     - “对比各部门业绩，做一个柱状图”
     - “各产品类别占比是多少，做一个饼图并显示百分比”
  3. 触发写回执行
- 期望结果：
  - 至少出现一个可见结果：排序/透视/图表/格式化（以 Plan 实际执行为准）
  - 不写错工作表/不写错文档
- 证据：按“成功/失败模板”留

### Scenario: WPP 典型生成（每轮建议跑）

> 对应 `TODOLIST_PPT.md` 的典型场景（产品介绍/项目汇报）。验收点是：能生成多页结构化 PPT，不会白屏。

- 步骤：
  1. 打开一个 WPP 演示文稿
  2. 发起对话（例如“做一个 8 页产品介绍 PPT：封面/目录/痛点/方案/功能/优势/案例/结尾”）
  3. 触发写回执行
- 期望结果：
  - 生成多页并写入标题/要点（允许 best-effort）
  - 不崩溃/不白屏；如果某一步失败，至少留一个可见占位结果并提示原因
- 证据：按“成功/失败模板”留

### Scenario: 4 文档并发（每轮建议跑；版本里程碑必跑）

- 步骤：
  1. 同时打开 2 个 Writer + 1 个 ET + 1 个 WPP
  2. 在两个 Writer 文档同时发起对话并流式返回
  3. 分别触发写回，让写回进入各自队列
- 期望结果：
  - 两个 Writer 都能收到各自的回复
  - 写回不会写错到另一个文档
  - 写回队列可见、可取消、失败有原因
- 证据：
  - 截图（至少包含“各文档对应的对话/写回状态”）
  - 若出现串台：记录 docKey/docId、相关 trace_id

### Scenario: 切文档不阻塞（每轮建议跑）

- 步骤：
  1. 打开两个 Writer 文档（A、B）
  2. 在 A 发起对话，让它处于“生成中”
  3. 立刻切到 B，观察历史是否立刻加载；并在 B 再发起一次对话
- 期望结果：
  - B 的历史立刻显示（不需要等 A 结束）
  - B 可以正常发起对话并返回
- 证据：截图（最好包含 A/B 两个文档各自的状态）

### Scenario: 取消生成 + 取消待写回（每轮建议跑）

- 步骤：
  1. 发起一次对话，在生成中点击取消
  2. 再触发一次写回，让它进入队列；在真正执行前取消该条写回
- 期望结果：
  - 取消后 UI 状态恢复正常（可以继续对话）
  - 被取消的写回不会继续执行、不会写进文档
- 证据：截图 +（若失败）对应日志片段

### Scenario: 远端后端（每个迭代周期至少跑一次）

- 步骤：
  1. 插件连接远端后端（云端/局域网均可）
  2. 任选一个宿主跑一次“发起对话 → 写回”
- 期望结果：
  - 不依赖后端本机路径（不会把“服务器上的路径”当作客户端文件路径）
  - doc snapshot / asset 等能力在远端仍可工作（至少不报明显的路径类错误）
- 证据：
  - 成功/失败截图 + 后端日志片段

### Scenario: 多模态生成图片（每个迭代周期至少跑一次）

> 目的：避免把大段 base64 图片塞进对话，导致前端重载/白屏；同时确认“生成图片 → 落 asset → 写回可用”这条链路可用。

- 步骤（两选一，哪个顺手用哪个）：
  1) 对话入口（如果当前 UI/技能已接通）：在 WPP 里发起对话，例如“做 1 页封面并配一张商务插图”，并触发写回
  2) 接口入口（如果暂时没 UI 入口）：调用一次 `POST /agentic/mm/generate-image`，确认返回 `asset_id`
- 期望结果：
  - 返回的是 `asset_id`（或等价的 asset 引用），而不是把整张图 base64 直接塞进对话内容
  - 在 WPP 侧能 best-effort 把 `asset://<id>` 插入到幻灯片；失败也要给出可见的占位结果与原因
- 证据：
  - 成功/失败截图（包含插图或占位形状）+ 后端日志片段（最好带 trace_id）

### Scenario: Writer/ET 插图（每个迭代周期至少跑一次）

> 目的：把“生成图片 → asset:// → 写回插图”的闭环补齐到 Writer/ET（不再只有 WPP 能插图）。

- 步骤（任选一个宿主跑通即可；里程碑建议两端都跑一次）：
  1. 打开 Writer 或 ET
  2. 触发一次“生成图片”（对话入口或 `POST /agentic/mm/generate-image` 均可），拿到 `asset_id`
  3. 触发一次写回（Plan）包含 `insert_image`，并把 `path` 设为 `asset://<asset_id>`
- 期望结果：
  - Writer：图片出现在光标处（best-effort）
  - ET：图片出现在当前选区附近（best-effort）
  - 如果插图失败：也不能把整次写回搞崩；必须有可见占位（例如写入一段“图片未插入”的提示）+ 可定位原因
  - 插入成功后：前端会 best-effort 调用 `DELETE /agentic/assets/{asset_id}` 清理临时资产（避免堆积）
- 证据：
  - 成功/失败截图（包含插图或占位）+ 后端日志片段（最好带 trace_id）

### Scenario: 两个租户互不影响（里程碑必跑）

> 目的：避免“串租”（A 租户能看到/读到/改到 B 租户的数据）。

- 步骤：
  1. 用 tenant A：上传一次 doc snapshot，并生成一次图片 asset（或任意会落到 assets 的动作）
  2. 用 tenant B：尝试访问 tenant A 的 snapshot/asset（用同样的接口/同样的 id）
- 期望结果：
  - 访问不到（返回 404 或 403）
  - 后端日志能定位到这次访问（最好能看到 trace_id/tenant_id）
- 证据：
  - 请求/响应截图或日志片段（能看出 tenant A/B）

### Scenario: RAG 串租验证（里程碑必跑）

- 步骤：
  1. 用 tenant A：把一段明显的内容写入 RAG（例如“ONLY_FOR_TENANT_A_123”）
  2. 用 tenant B：查询这段内容
- 期望结果：
  - tenant B 查不到 tenant A 的内容
- 证据：
  - 查询截图（或返回结果）+ 后端日志片段

### Scenario: SSRF 黑名单能拦住并留审计（里程碑必跑）

> 目的：当租户配置了黑名单，后端出网能力会被拦住，并且能留下审计记录，方便复盘。

- 步骤：
  1. 在租户的 SSRF 黑名单里加入一条规则（例如拦 localhost/metadata/私网段/或“重定向到私网”的情况）
  2. 触发一次后端出网（例如网页抓取/浏览器快照/URL 入库）
- 期望结果：
  - 该请求被拦住（失败原因清楚）
  - 审计里能看到这次 allow/deny 记录（至少包含 tenant_id、trace_id、reason）
- 证据：
  - 审计文件片段（脱敏）或截图 + 后端日志片段

### Scenario: 连续对话 10 轮不重载（里程碑必跑）

- 步骤：
  1. 在同一个文档里连续对话 10 轮
  2. 其中至少 1 轮包含 doc snapshot 的上传/解析路径（例如打开一个内容较多的文档，让后端走 snapshot 机制）
- 期望结果：
  - 前端不因为“文档上下文传输/解析”导致重载或白屏
  - 对话里不会出现“把整篇正文/整张图 base64 直接塞进消息”的情况（避免 payload 过大）
  - 过程中即使失败，也能继续下一轮（不需要重开插件才能继续）
- 证据：
  - 前端状态截图（最好含 boot-status/错误覆盖层信息）

### Scenario: 任意失败都能定位原因（每轮建议跑）

- 步骤：
  1. 触发一次明确的失败（例如填错 URL/用错租户 key/让写回故意失败一次）
- 期望结果：
  - 用户能看到“失败原因是什么、下一步怎么做”
  - 后端日志里能按 trace_id/tenant_id 查到堆栈与原因
- 证据：
  - 错误提示截图 + 对应后端日志片段
