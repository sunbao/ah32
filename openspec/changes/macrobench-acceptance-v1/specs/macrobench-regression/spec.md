# Spec — macrobench-regression (v1)

## 一句话说明

这份规范定义“怎么做回归长跑”：用一套固定的跑法，快速发现“写回成功率掉了/某宿主崩了/远端模式坏了”。

## 目标

- 让每轮改动后都能用同一套方式做回归，不靠“感觉差不多”。
- 失败时能复盘：知道哪一步坏了、怎么复现、需要看哪段日志。

## 最小跑法（v1）

### Requirement: 每轮最小回归（必须做）

- 每轮至少跑：
  - Writer 1 条用例
  - ET 或 WPP 至少 1 条用例
- 目的：避免只在一个宿主上“自嗨式通过”。

### Requirement: 每个迭代周期做一次三端回归（建议）

- 每个迭代周期（例如你觉得功能稳定到可以验收的一次）至少跑：
  - Writer 1 条 + ET 1 条 + WPP 1 条
- 目的：保证三端都没被带崩。

### Requirement: 失败必须能复现（留最小材料）

每次失败至少要能留下这些信息（不要求进 git，但要能找得到）：

- 用户输入（或用例描述）
- 当时选用的宿主（wps/et/wpp）与目标文档身份（docId/docKey）
- 写回产物（Plan JSON 或最终执行产物的摘要）
- 错误提示（前端看到的那句话）
- 后端日志片段（最好能带 trace_id/tenant_id/user_id）

## 通过/失败判定（v1）

### Requirement: 成功的定义

一次用例算“成功”，需要满足：

- 对话能返回（流式或非流式都行）
- 触发写回后文档发生了预期变化（有可见结果）
- 过程中没有把内部 JSON/调试内容写进文档

### Requirement: 失败要分类型

失败要能分出大概是哪类问题，避免都叫“失败”：

- 网络/连接类（连不上、超时、断流）
- 生成类（没有生成可执行 Plan、输出不符合 schema）
- 执行类（Plan 合法但执行失败、宿主 API 差异）
- 串台类（写错文档/写错宿主）

## 建议的跑测记录格式（人看得懂）

每次跑测建议记录成下面这种“人话结果”，方便你审核：

- 本轮改了什么（1 句话）
- 跑了哪些宿主（Writer/ET/WPP/远端后端）
- 成功/失败各多少
- Top 失败原因（1-3 条）
- 失败复现材料放哪（本地路径/截图/日志）

## 证据模板（跑完就按这个交付）

> 目的：你看到这份记录就能验收，不需要再追问一堆“你怎么跑的”。\n

- 跑测时间：YYYY-MM-DD
- 版本/提交号：例如 `git rev-parse --short HEAD`
- 跑测范围：
  - 每轮最小回归：Writer 1 条 + ET/WPP 1 条（是/否）
  - 三端回归：Writer/ET/WPP 各 1 条（是/否）
  - 远端后端：跑了哪个宿主（无/Writer/ET/WPP）
- 结果汇总：
  - 成功：X
  - 失败：Y
  - 主要失败原因：1/2/3
- 复现材料：
  - 失败用例编号/描述
  - 对应截图/日志位置
  - 对应 trace_id（若有）

## 工具与观测（v1）

### Requirement: 有一键“生成质量回归”（不依赖 WPS）

> 目的：很多回归问题不是 WPS 执行崩，而是模型没输出可执行 Plan。这个用脚本就能先拦住。

- 用法：用 `scripts/macro_bench.py` 跑一组用例，统计：
  - 成功率
  - 尝试次数（内部重试次数）
  - 延迟
  - 输出的 op 分布（大概是什么操作）
- 判定：如果出现“Plan 为空/不是 ah32.plan.v1/一直报 schema 错”，直接算回归失败。

### Requirement: 能把关键事件记录下来（便于复盘）

> 目的：失败时别靠猜，至少能知道“卡在哪一步”。

- 前端会把关键事件上报到后端（best-effort）：例如开始生成、开始写回、写回成功/失败、失败原因等。
- 后端提供最小接收口：
  - `POST /telemetry/events`（接收事件）
  - `GET /telemetry/capabilities`（告诉你当前有没有开、怎么存）
- 判定：开了 telemetry 时，事件能被接收；没开时，不影响主流程（只是没有观测数据）。

### Requirement: 能看到“本轮命中的技能”（便于判断走没走对）

- 前端聊天面板会展示本轮技能信息：
  - “选”：后端路由选中的 skills（本轮注入）
  - “用”：输出中后验检测到的 skills（本轮实际用到）
- 判定：做一轮对话时，至少能看到“命中/未命中”的提示，不是完全黑盒。

### Requirement: 回归用例要带“风格参数”（避免每次风格飘）

> 目的：同一个场景，今天输出“像样”，明天输出“糊”，很难回归；风格参数能让输出更可控。

- 回归用例/长跑用例可以携带一个 StyleSpec（例如“商务蓝”“标题层级”“表格边框”等偏好）。
- 后端需要能接收并做最小纠错（字段缺省/大小写/宿主差异），再注入到生成提示里。
- 判定：
  - 用例里带 styleSpec 时，后端不会报错（能正常跑通生成）
  - 生成结果至少不会完全无视（具体好不好看仍以手工验收为准）

### Requirement: 动态技能选择可开可关（避免“技能乱入”）

> 目的：有时候你想要“智能选技能”，有时候你只想要“别折腾、稳定输出”。这得能一键切换。

- 支持开关动态技能选择：
  - 开：每轮根据输入挑 0..N 个技能注入
  - 关：不做动态选择（或退化为固定注入/不注入）
- 同时要有“上限”：
  - 本轮最多选几个技能
  - 技能注入总字符预算（超了要截断/降级）
- 判定：
  - 关闭后，本轮不再出现“选中的技能列表”
  - 开启后，本轮能看到“选中的技能列表”，并且不会无限塞爆上下文

### Requirement: RAG 命中情况要能看（别完全靠猜）

> 目的：你问“为什么这次答得对/为什么答偏了”，至少要能看见有没有检索、检索了多少、有没有被截断。

- 开启调试信息时，可以看到本轮 RAG 的命中摘要（命中条数/来源预览等）。
- 同时后端会把 RAG 的检索信息记到 telemetry（best-effort），用于回归对比。
- 判定：
  - 开启调试后能看到 RAG 摘要；关闭调试后不影响正常聊天/写回
