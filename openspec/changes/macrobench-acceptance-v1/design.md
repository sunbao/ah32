# Design: macrobench-acceptance-v1

## 目标（用大白话）

我们现在有一堆写在根目录 `TODOLIST*.md` 里的“需求 + 现状 + 怎么验收”。这套清单很好用，但它不在 OpenSpec 流程里，久了会出现：

- 谁决定的、何时验收的、验收证据在哪：对不上账。
- 验收动作分散在多个 change/清单里：容易漏。
- 发现问题后，不知道应该“改清单”还是“开新 change 改代码”：口径不统一。

所以这个 change 的目标是：把这些清单收口到 OpenSpec 里，让 OpenSpec 接管“验收怎么跑、跑完怎么判定、证据怎么留”，最终可以删除根目录 `TODOLIST*.md`。

## 不做什么（避免跑偏）

- 不在这个 change 里强行做功能开发（如果验收时发现缺功能/不合理，另开 follow-up change）。
- 不追求一次把所有事情都“写成很学术的规范”。先把“能按图跑验收”写清楚。
- 研发阶段不兼容旧客户端，这里只按“当前版本前后端配套”来写验收。

## 输入材料（我们要收口的东西）

- 根目录：`TODOLIST.md`、`TODOLIST_WRITER.md`、`TODOLIST_ET.md`、`TODOLIST_PPT.md`
- 现有 OpenSpec changes 的“手工验收项”（例如 remote-backend、租户隔离、SSRF 审计等 change 里 4.x 这种条目）
- 现有代码的“实际能力”：以代码为准，文档与 tasks 只是跟踪（发现不一致要补回到 OpenSpec）

## 输出物（这个 change 要产出什么）

本 change 只产出“验收/回归的规范与清单”，让后续每轮迭代都有统一跑法：

1) `office-manual-acceptance`（手工验收矩阵）
- 覆盖 Writer / ET / WPP
- 覆盖“本机后端”和“远端后端”两种部署

2) `macrobench-regression`（长跑回归跑法）
- 怎么跑、跑多久、怎么判定通过/失败
- 需要留哪些证据（截图/日志片段/trace_id 等）

3) 对应的 `tasks.md` / `review.md`
- tasks：把“要补齐的验收条款/映射关系”列成可勾选任务
- review：记录每次收口的结论（哪些确认可删、哪些要开新 change）

## 核心规则（收口时怎么做决定）

### 1）“清单条目”分三类

把 `TODOLIST*` 里的每条内容按下面三类归档，避免混在一起：

- **必须每轮都跑的验收**：例如“真环境至少跑一次（Writer/ET/WPP）”“同时打开多个文档/多个宿主不崩”等。
- **某次改动触发才需要跑的验收**：例如“改了 doc snapshot，就重点跑 doc snapshot 的远端后端分支”。
- **信息/现状说明**：例如“当前已支持什么、未来愿景是什么”（这类不属于验收条款，应该进说明文档或 review 结论里）。

### 2）验收必须留“最小证据”

验收不是写一句“我跑过了”就算。每条验收至少留一条最小证据，方便复盘：

- **成功**：截图（含 UI 状态/关键结果）+ 关键日志（最好能定位到 trace_id/tenant_id/user_id）
- **失败**：截图 + 错误提示 + 后端日志片段（堆栈）+ 触发步骤（1/2/3）

证据不强制进 git（可能包含敏感信息）；可以存在本地/私有 QA 目录，但验收条款里要写清楚“证据长什么样、去哪找”。

### 3）发现不合理/缺功能：不要在这里硬塞

收口时经常会遇到两类情况：

- **文档写的跟代码不一样**：那就以代码为准，把 OpenSpec 文档改成真实情况（并在 review 里写清楚“为什么改”）。
- **确实缺功能/需要改代码**：不要在本 change 里混着改，应该开一个新的 follow-up change，写清楚需求与验收。

这个 change 的职责是“把验收口径收拢”，不是“顺手把所有问题都修了”。

## 验收矩阵的结构（后面 specs 会按这个来写）

每个验收条目建议有这些字段（写人话即可）：

- **场景**：我在做什么（例如“Writer 写回插图”“远端后端抓取网页入库”）
- **前提**：需要什么环境（例如“远端后端模式”“登录/不登录”“是否有租户”）
- **步骤**：1/2/3 怎么点怎么做
- **期望结果**：用户能看到什么（例如“文档出现图片”“不出现 Plan JSON”“不会白屏”）
- **失败时怎么看**：去哪里找日志/trace_id，截图要截哪里

## “删 TODOLIST*.md” 的条件（避免删早了）

根目录 `TODOLIST*.md` 只有在满足这些条件后才允许删除：

1) `office-manual-acceptance` + `macrobench-regression` 已覆盖 `TODOLIST*.md` 的核心验收项（至少是“必须每轮跑”的那一批）
2) 已把 “现有 change 里的手工验收项” 统一收进验收矩阵（不再分散）
3) review 里写清楚：哪些条目被合并/改写了，为什么

删除动作单独一个提交做，方便回滚与审查。

## 后续工作（本 change 的下一步）

- 先补 `specs/office-manual-acceptance/spec.md` 与 `specs/macrobench-regression/spec.md`
- 再把 `TODOLIST*.md` 的条目逐条映射进来（映射表放到 tasks/review 中）
- 映射完成后，开一笔提交删除 `TODOLIST*.md`
