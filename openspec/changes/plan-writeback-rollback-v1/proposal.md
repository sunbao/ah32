# Proposal: plan-writeback-rollback-v1

## 为什么要做（人话）

现在项目主写回通道是 **Plan 写回**（`ah32.plan.v1`）。但旧的 `TODOLIST.md` 里写过一套“预览/应用/带回退应用（apply_with_backup）”的体验，看起来像是“已经做完了”。

问题是：这套“回退/版本块”的体验，**在 Plan 写回链路里并不完整**。很多能力其实是旧 JS 宏通道里做的，Plan 通道只覆盖了一部分（例如 Answer Mode 有 `backup` 参数）。

这会导致：

- 用户以为“写回后能一键回退”，结果在 Plan 写回里回退不了或不稳定
- 验收时口径混乱：到底以哪个通道为准？

所以需要把“回退/预览/版本块”这件事，**在 Plan 写回链路里补齐口径与实现**。

## 目标（v1）

1) 写回默认更安全：能回退（至少针对 Writer 的块级写回/答题写回）
2) 回退动作可用：用户点一下就能回退到上一版（不需要手工复制粘贴）
3) 失败好定位：失败时能看到原因，后端/前端有日志可查

## 不做什么（v1 先别做）

- 不追求“完整的 Word/WPS 修订模式”能力（那是另一套产品）
- 不做复杂的多版本管理中心（先解决“上一版能回退”）

## 需要补的能力（大方向）

### 1) 写回时自动留一份“上一版”

- 对 `upsert_block` / `answer_mode_apply` 这类会改文档的动作，默认在执行前把“旧内容”存一份（按 doc + block_id）

### 2) 有一个“回退”入口

- UI 上能对某个写回产物点“回退”
- 回退成功/失败都有明确提示

### 3) 预览模式（可选，但建议纳入）

- 有些场景（合同/制度）用户更希望先看对照表，再决定是否应用
- v1 可以先把“预览”作为一种写回策略（不强行一键覆盖）

## 与现有 changes 的关系

- `macrobench-acceptance-v1`：验收矩阵里会要求“失败可定位/写回安全默认策略”。本 change 是把“能回退”补成真正可用。
- `spec-implementation-audit`：这里是被点名的“差异点/冲突点”，已记录为 MUST-FIX。

## 需要你确认的问题（开工前必须定）

1) “回退”只做 Writer（WPS 文档）够不够？ET/WPP 暂时不做可以吗？
2) “回退”只回退到上一版够不够？还是要保留 N 版？
3) 预览模式你希望是什么体验：只在聊天里给对照表，还是也能写进文档但不应用？
