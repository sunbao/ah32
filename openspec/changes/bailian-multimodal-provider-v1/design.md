# Design (Draft) — Bailian Multimodal Provider (v1)

## Overview

This change introduces a provider-pluggable multimodal runtime (`analyze_image`, `generate_image`) with **Bailian** as the default provider.

## Architecture

- Provider interface: `MultimodalProvider`
- Provider selection: env-driven (`AH32_MM_PROVIDER=...`)
- Storage: generated bytes go to the ephemeral Asset Store (see `multimodal-writeback-assets-v1`)

## Interfaces

### Provider interface (internal)

- `analyze_image(prompt, image_bytes, mime, opts) -> {text, structured?}`
- `generate_image(prompt, opts) -> {asset_id, mime, bytes, provider, model}`

### Backend API

- `POST /agentic/mm/analyze-image`
- `POST /agentic/mm/generate-image`

## Error handling & strictness

- Default strict: missing keys/models => fail with clear error.
- No silent downgrade to other providers.

## Observability

- Log provider/model, request sizes, durations, trace_id (never log image bytes/prompt).
- Emit events for latency/failure (if telemetry exists).

## Security & privacy

- Asset bytes are ephemeral; delete after use/turn; TTL only as crash-safety.
- Never persist into `data/` or RAG.

## Open questions

1) Bailian API: do we have an existing SDK/client, or implement raw HTTP?
2) Input images: do we require `asset_id` only (preferred) and allow `data_url` only under a strict cap?
3) Frontend flow: should image-understanding also go through Asset store first to avoid large chat payloads?
