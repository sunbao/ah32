## Context

现状：

- 招投标相关对话场景依赖 RAG（向量检索）提供“合同/案例/模板/政策法规”等知识支撑，但这些数据往往尚未进入当前项目知识库。
- 目前系统具备底座能力：RAG 文本/文件导入接口、Playwright 浏览器控制层（抓取网页内容），以及技能（skill）提示词/能力声明体系。
- 但缺少端到端闭环与明确的行为契约：当 RAG 0 命中时，是否必须反馈“缺库并引导补库”；以及来源（source/URL）是否默认展示，容易破坏正文/写回排版。

约束：

- 本阶段仅覆盖《加快招标投标领域人工智能推广应用的实施意见》相关的招投标技能场景；不引入关键词检索，沿用向量检索。
- “缺库强制约束”属于技能级别的业务规则；架构（core）只提供可复用机制/信号，不把业务写死。
- 默认不在正文/写回里展示 source/URL；仅当用户明确要求“出处/来源/引用/链接”等时展示。
- 禁止吞错：出现异常必须可追踪（日志/事件/上报），并对用户给出可执行的下一步操作指引。

## Goals / Non-Goals

**Goals:**

- 统一并可配置地支持“RAG 0 命中缺库反馈”：
  - core 提供可复用的“RAG 0 命中提示机制/能力开关”；
  - 技能可声明强制启用，并在对话中给出补库路径（上传/导入、或 URL 抓取入库）。
- 建立“URL → 抓取正文 → 入库（项目库优先） → 可检索”的端到端闭环，确保用户补库后能在同一项目对话中命中。
- 建立“来源展示策略”能力：默认不输出 source/URL，按对话意图决定是否展示出处/链接（并避免污染写回内容）。
- 对抓取失败提供确定性的兜底：提示用户粘贴纯文本（暂不支持图片）。

**Non-Goals:**

- 不做关键词检索/混合检索策略的引入与调参（后续按真实使用效果再评估）。
- 不承诺当前版本内置就具备完整“合同库/案例库/模板库”的数据集；重点是流程增强与缺口反馈机制。
- 不在本变更中覆盖所有 20 个场景的完整业务自动化；只实现当前实施意见驱动的“缺库反馈 + 补库操作闭环 + 来源展示策略”这类横切能力。

## Decisions

### 1) 缺库约束：技能决定，core 提供机制

**Decision：** core 不直接强制“RAG 0 命中一定要提示缺库”，而是通过 skill capability（例如 `capabilities.rag_missing_hint=true`）让技能声明是否启用；当启用时，core 在上下文中注入“RAG 0 命中提示信号”，技能必须据此触发缺库反馈与补库指引。

**Alternatives considered：**

- A) core 全局强制缺库提示：实现简单，但会污染非招投标技能、并造成“所有问答都强制引导导入”的反效果。
- B) 完全由技能自行判断（core 不提供信号）：会导致不同技能实现不一致，且难以在路由/编排层统一处理“0 命中”的确定性反馈。

### 2) 来源展示策略：默认隐藏，仅按用户意图展示

**Decision：** 将“是否展示出处/链接”视为对话策略能力：默认不展示 source/URL，只有当用户明确要求（如“给出处/链接/引用”）才展示；并要求写回（Plan JSON 写回正文块）同样遵守该策略，避免 source 破坏排版。

**Alternatives considered：**

- A) 永远展示引用：可追溯性更强，但正文排版被污染，且用户常需要手工删除。
- B) 永不展示引用：影响合规与可追溯需求；不满足部分“必须提供出处”的用户场景。

### 3) URL 入库闭环：优先项目库；失败兜底为纯文本

**Decision：** 在招投标技能中提供一个明确的“URL 抓取入库”工具/流程：

- 输入：URL +（可选）文档标题/分类（合同/案例/模板/政策等）。
- 抓取：使用现有浏览器控制层抓取正文文本；检测 captcha/空白/不可达等失败类型。
- 入库：调用现有 RAG 文本上传/导入能力，将抓取结果写入向量库；并在存在项目上下文时，默认挂到“项目 RAG 索引/项目库”。
- 兜底：抓取失败时，提示用户复制粘贴纯文本（当前不支持图片），并提供“上传文件/导入目录”的操作路径。

**Project scoping 规则：**

- 有项目上下文（例如存在 `contextDocumentPath` / 项目 id）→ 默认进入该项目可检索范围。
- 无项目上下文 → 进入全局库或提示用户选择导入范围（后续可在 specs 中固化规则）。

**Alternatives considered：**

- A) 入库一律全局：实现最简单，但容易跨项目污染检索结果，且不符合“项目库”预期。
- B) 一律项目库：在无项目上下文时无法落地，需要额外交互或 fallback。

### 4) 向量检索优先：不引入关键词检索

**Decision：** 本阶段仅使用现有向量检索链路；任何“检索不到”都走缺库反馈 + 补库闭环，而非切换关键词检索兜底。

## Risks / Trade-offs

- [网页抓取不稳定（登录/反爬/captcha/动态渲染）] → 提供可诊断失败原因 + 明确兜底（粘贴纯文本/上传文件）；抓取流程全程记录日志与 trace_id。
- [来源默认隐藏降低可追溯性] → 当用户明确要求出处时再展示；并在内部元数据保存来源信息（不污染正文）。
- [项目库归属错误导致“入库了但检索不到”】【例如未加入项目索引】] → 入库接口必须返回可用于检索过滤的 source_id，并在项目索引层原子更新；失败要对用户反馈“入库未完成/需重试”。
- [技能提示词与 core 机制耦合过深] → 用 capability/契约定义边界（specs 中固化），core 只负责信号与基础设施。

## Migration Plan

1. 先落地能力契约（specs）与技能侧提示词（缺库反馈、来源策略、URL 入库引导）。
2. 新增/完善 URL 抓取入库工具链，接入现有 RAG 导入 API，并支持项目索引更新。
3. 端到端回归验证：
   - RAG 0 命中必须触发缺库提示与补库步骤；
   - URL/文本导入后可在同一项目内检索命中；
   - 未要求时不展示来源；要求时才展示来源。
4. 灰度发布（如有）：先仅在招投标技能启用 capability，再逐步扩展到其他技能。

Rollback：

- 保留 capability 开关；出现问题可临时关闭 `rag-missing-library-guidance`/`rag-url-ingest` 对应能力或工具入口，不影响核心对话与基础 RAG 功能。

## Open Questions

- “项目库”判定的唯一信号来源：以 `contextDocumentPath`、会话 project_id，还是用户显式选择为准？
- 来源展示的“明确要求”判定规则：仅关键词触发，还是提供 UI 开关/显式参数？
- URL 抓取的正文抽取策略与长度限制：如何在保证检索效果的同时避免噪声与超长文本入库？
